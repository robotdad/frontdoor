# Frontdoor Hosting Guide

This guide covers TLS provisioning, FQDN detection, app integration, cookie behavior, and uninstalling.

---

## TLS Provisioning

Frontdoor's installer detects the best available TLS tier at install time and configures Caddy accordingly. The three tiers in priority order:

| Tier | Method | Cert Path | When Used |
|------|--------|-----------|-----------|
| 1 — Tailscale | `tailscale cert` | `/etc/ssl/tailscale/` | Tailscale is installed and the machine has a Tailscale DNS name |
| 2 — Self-signed | `openssl req` | `/etc/ssl/self-signed/` | Tailscale is unavailable; installer generates a self-signed cert automatically |
| 3 — Plain HTTP | _(no cert)_ | _(none)_ | Neither Tailscale nor openssl is usable; only acceptable on a trusted LAN |

**Self-signed cert details:** RSA 2048-bit keys with 10-year validity (`-days 3650`). Generated once by the installer; renewed manually if needed.

> **Plain HTTP note:** Plain HTTP should only be used on a Tailscale tailnet or another trusted LAN where traffic is otherwise protected. It is the ultimate fallback, not the recommended path.

## FQDN Detection

The installer determines the machine's Fully Qualified Domain Name (FQDN) using this priority:

1. **Tailscale DNS name** (preferred) — obtained via `tailscale status --json` if Tailscale is running. Example: `myhost.tail1234.ts.net`.
2. **`hostname -f` fallback** — used when Tailscale is unavailable or returns no DNS name.

The detected FQDN is used for:

- **Caddy vhost** — the `*.ts.net` (or fallback) hostname in the Caddy site block.
- **Cert CN** — the Common Name field in any self-signed cert generated by the installer.
- **Cookie domain** — the domain on which the `frontdoor_session` cookie is set, enabling shared auth across all apps on the host.

> **LAN without DNS note:** On a LAN without DNS, `hostname -f` may return only a short hostname (e.g., `myhost`) or fail entirely. In this case, set `FRONTDOOR_COOKIE_DOMAIN` explicitly in `frontdoor.env` and ensure your browser can resolve the hostname.

## How Frontdoor Integrates with Apps

Every request to an app behind frontdoor flows through Caddy's `forward_auth` middleware, which calls frontdoor to validate the session cookie before allowing the request through.

```
Browser
  │
  │  HTTPS request
  ▼
Caddy
  │
  │  forward_auth (every request)
  ▼
frontdoor /api/auth/validate
  │
  │  validates cookie (frontdoor_session)
  │
  ├─ cookie invalid → 401 → Caddy redirects to frontdoor login
  │
  └─ cookie valid → 200 + X-Authenticated-User header
         │
         ▼
       Caddy reverse_proxy
         │
         │  X-Authenticated-User: <username>
         ▼
       App (receives authenticated request)
```

The `X-Authenticated-User` header contains the verified username. Apps read it directly — no separate auth flow required.

## Putting an App Behind Frontdoor

Short checklist — do all six steps:

1. **Bind to localhost** — configure your app to listen on `127.0.0.1:<port>`, not `0.0.0.0`.
2. **No app TLS** — Caddy handles TLS termination; your app serves plain HTTP internally.
3. **No app auth** — frontdoor's `forward_auth` gate ensures only authenticated requests reach your app.
4. **Read `X-Authenticated-User`** — access the verified username from the request header:
   ```python
   user = request.headers.get("X-Authenticated-User", "unknown")
   ```
5. **Write a Caddy snippet** — create `/etc/caddy/conf.d/<appname>.caddy` with `forward_auth` and `reverse_proxy` directives (use the `web-app-setup` skill or the `app.caddy.template` in this repo).
6. **Drop a manifest** (optional) — add `/opt/frontdoor/manifests/<appname>.json` to customize the dashboard display name, description, and icon.

**amplifierd note:** If your app is built on `amplifierd`, set `AMPLIFIERD_TRUST_PROXY_AUTH=true` in its environment to enable header-based identity without a separate login flow.

## Cookie Behavior

Frontdoor uses a single `frontdoor_session` cookie to track authenticated sessions.

| Variable | Value | Condition |
|----------|-------|-----------|
| `FRONTDOOR_SECURE_COOKIES` | `true` | HTTPS is active (Tailscale cert or self-signed cert) |
| `FRONTDOOR_SECURE_COOKIES` | `false` | Plain HTTP fallback only |
| `FRONTDOOR_COOKIE_DOMAIN` | _(FQDN, auto-detected)_ | Set automatically at install time; override to change cookie scope |

> **Changing these values:** Edit `/opt/frontdoor/frontdoor.env` and restart the service (`systemctl restart frontdoor`). The installer sets these variables based on the detected TLS tier; manual overrides persist across restarts.

## Uninstalling

**Basic uninstall** — stops the service, disables it, and removes the systemd unit. Application data in `/opt/frontdoor` is preserved.

```bash
sudo deploy/uninstall.sh
```

**Purge** — also removes `/opt/frontdoor` and the self-signed certificates in `/etc/ssl/self-signed/`.

```bash
sudo deploy/uninstall.sh --purge
```

> **Not removed by either command:** Caddy itself, Tailscale certificates (`/etc/ssl/tailscale/`), Caddy `conf.d` snippets, and the main `Caddyfile`. These may be shared with other services.

## Environment Variables

| Variable | Description |
|----------|-------------|
| `FRONTDOOR_SECRET_KEY` | Secret key used to sign session cookies. Must be kept secret; changing it invalidates all active sessions. |
| `FRONTDOOR_SECURE_COOKIES` | `true` to set the `Secure` flag on the session cookie (required for HTTPS). `false` for plain HTTP only. |
| `FRONTDOOR_COOKIE_DOMAIN` | Domain on which the `frontdoor_session` cookie is set. Defaults to the detected FQDN. Set explicitly for LAN hosts without DNS. |

All variables live in `/opt/frontdoor/frontdoor.env`. Restart frontdoor after any change:

```bash
sudo systemctl restart frontdoor
```
